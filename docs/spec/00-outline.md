## Original outline (planning notes)

> **What this is:** the annotated table of contents for the Ebrostay
> **reconstruction specification** — a document set complete enough to rebuild the
> entire system from scratch **without the source code**: every page, every filter,
> every field, every business rule, every integration, and every decision taken
> (with its rationale).
>

## §1 — Overview & goals

- **Purpose:** what Ebrostay is, the problem it solves, the business model
  (mid-term 1–11 month furnished rentals; no online payment; manual confirmation).
- **Contents:** product summary; the two core principles (graceful degradation;
  no online payment); scope boundaries (what the system does / explicitly does not
  do); target users (tenants, owners, operators) and what each needs.
- **Depth bar:** a new engineer/PM understands the product and its constraints
  without prior context.
- **Sources:** README, `spotahome-inspired-sdd.md` §3–5, requirements.md §1.

## §2 — Glossary & domain concepts

- **Purpose:** define the vocabulary precisely so every later section can use terms
  unambiguously.
- **Contents:** stay, booking **request** vs booking, availability **block** vs
  **hold**, commission, deposit, upfront rent, billed months, `bills_policy`
  (included/capped/excluded), property `type`, owner vs admin, published vs draft.
  Each term: definition + where it appears in data/UI.
- **Depth bar:** no term used later is undefined here.
- **Sources:** synthesized from code + data dictionary.

## §3 — Architecture

- **Purpose:** the shape of the system, enough to stand up an equivalent.
- **Contents:** static front end (no build step) + optional Supabase backend;
  component diagram; data-flow (browser ↔ Supabase REST/Auth/Storage ↔ Edge
  Functions ↔ Resend/DeepSeek); the **graceful-degradation** mechanism (`backend.js
  isConfigured()` → sample-data fallback); configuration (`supabase-config.js`,
  `.env`, `scripts/inject-config.js`); hosting & deploy (GitHub Pages + Actions,
  custom domain, DNS); file/asset inventory (what each top-level JS/HTML file owns).
- **Depth bar:** a reader can recreate the project skeleton, config, and deploy
  pipeline.
- **Sources:** README, `supabase-setup.md`, `.github/workflows`, package.json.

## §4 — Data model

- **Purpose:** every table, column, type, constraint, default, and relationship.
- **Contents:** all 11 tables + storage bucket + auth touchpoints, field-by-field
  with examples; ER/relationship overview; constraints (overlap exclusion, FKs,
  checks); per-table RLS summary (full policy text lives in §8). Status banners on
  `bookings` (🗑️) and `booking_requests` (🔜).
- **Depth bar:** the schema can be recreated exactly (a `schema.sql` could be
  written from this section alone).
- **Sources:** **`data-dictionary.md` migrates here wholesale.**

## §5 — Business rules & algorithms

- **Purpose:** the logic — the part that's easiest to get wrong on a rebuild and
  that prose specs usually fumble. Each rule as **plain-language + pseudocode +
  worked example(s)**.
- **Contents:**
  - **Pricing:** billed-months calculation (whole months, round-up, min 1, end
    exclusive; the 30 vs 31-day cases); rent = months × price; commission = min(15%,
    one month) + the "commission discount" presentation; deposit; total. Both the
    client (`property.js`) computation and the server (`request-booking`) one, with a
    **parity note**.
  - **Filtering:** the exact predicate a property must satisfy (type, budget,
    guests, amenities, available-from, date-overlap) + enhanced filters (address
    search fields, min beds/baths, saved-only 🚫 out-of-scope (MVP)).
  - **Sorting:** `best` / `price` / `new` ordering keys.
  - **Availability:** block vs hold semantics; overlap test (inclusive); expired-hold
    filtering; the GiST no-overlap constraint.
  - **Validation rules:** date ranges, min/max stay, required fields per form.
- **Depth bar:** every numeric output in the UI is reproducible to the cent/day from
  this section.
- **Sources:** `property.js`, `request-booking/index.ts`, `site.js`, `enhance.js`,
  requirements.md R-Prop-5/-6/-7, R-Home-3/-5/-6.

## §6 — Functional specification (per page)

- **Purpose:** the exhaustive "what every screen does" — every control, state, and
  message, in **both languages**.
- **Structure:** one subsection per page; each documents Layout → Controls (every
  input, its type/validation/default) → States (default, loading, empty, error,
  success, not-found) → Actions (what each button does) → Copy (ES/EN strings or
  keys).
  - **6.1 Home** (`index.html`/`site.js`): hero search, filter panel, quick filters,
    sort, listings grid + card anatomy, map panel, inquiry form, auth dialog, header/
    footer/nav, owner-mode nav.
  - **6.2 Property detail** (`property.html`/`property.js`): gallery/lightbox, video,
    floor plans, conditions table, move-in cost, location map, availability calendar,
    booking widget (the mailto/WhatsApp MVP flow), share, not-found.
  - **6.3 Account** (`account.html`/`account.js`): signed-out/in, stays list,
    deactivation.
  - **6.4 Partner/owner** (`partner.html`/`partner.js`): routing by role, metrics,
    properties, payout form.
  - **6.5 Admin** (`admin.html`/`admin.js`): access gate, properties/requests/
    bookings/users tabs.
  - **6.6 Admin property editor** (`admin-property.html`/`admin-property.js`):
    details form, photos/floorplans, geocoding, AI assistant, guest info,
    availability blocks.
  - **6.7 Static pages:** about, privacy, 404.
- **Depth bar:** a developer can rebuild each screen, including every edge state and
  the exact copy, without seeing the originals.
- **Sources:** all HTML/JS, requirements.md §3, the explore-agent inventories already
  gathered.

## §7 — Integrations

- **Purpose:** every external dependency and how it's wired.
- **Contents:** Supabase (Auth incl. providers/SSO, REST, Storage bucket, Edge
  Functions); **Resend** email (the team + customer templates, env: `RESEND_API_KEY`,
  `EMAIL_FROM`, `EMAIL_TO`) — note 🔜 currently unwired; **DeepSeek** AI assistant
  (`ai-property-assistant`, `DEEPSEEK_API_KEY`, extract/translate/describe contracts);
  **Nominatim** geocoding; **Leaflet + OpenStreetMap** tiles; **flatpickr**;
  **Tesseract.js** OCR; **Umami** analytics. Each: purpose, config/secret, request/
  response shape, failure behavior.
- **Depth bar:** each integration can be re-provisioned and re-wired from this
  section.
- **Sources:** `deepseek-ai-setup.md`, both edge functions, backend.js, README.

## §8 — Auth & security model

- **Purpose:** the complete access-control picture — the only real security boundary.
- **Contents:** roles & how they're assigned; the public-anon-key model; **full RLS
  policy text** for every table + storage; the `security definer` RPCs and their
  guards; the client-gate-vs-server-gate distinction (JS `getIsAdmin()` is cosmetic);
  signup trigger; deactivation vs deletion; threat notes (e.g. world-readable
  `user_id` 🐞).
- **Depth bar:** every policy and RPC can be recreated; a security reviewer can
  audit from this alone.
- **Sources:** schema.sql, all `upgrade-*.sql`, data-dictionary RLS sections.

  (dark/light, storage, OS fallback); SEO/JSON-LD/sitemap/robots/llms.txt; PWA

- **Depth bar:** a rebuild matches formatting, translations behavior, SEO and a11y

- **Sources:** `style-guide-audit.md`, data.js, the HTML heads, styles.css.

## §10 — Acceptance criteria / test catalogue

- **Purpose:** the verifiable pass/fail conditions — the bridge to the E2E test plan.
- **Contents:** the tagged, AC-bearing requirements (✅/🔜/🗑️/🐞) grouped by area;
  the coverage map; explicit RLS negative-test cases.
- **Depth bar:** a tester can derive the full E2E plan directly.
- **Sources:** **`requirements.md` migrates here** (its AC content).

## §11 — Decision log (ADRs)

- **Purpose:** "every decision taken" + the **why** — the piece ordinary specs omit
  and the thing that makes a rebuild faithful to intent, not just to code.
- **Contents:** one entry per significant decision: context → decision → rationale →
  consequences → status. Seed entries already surfaced:
  - No online payment / Stripe removed; mailto-WhatsApp MVP booking flow.
  - Edge Function booking path built but left unwired (planned).
  - Commission = 15% capped at one month's rent.
  - Billed in whole months, end-date exclusive.
  - Security via RLS + public anon key (no app server).
  - Deactivation instead of deletion; admin-only hard delete.
  - `bills_policy` supersedes the legacy `bills_included` flag.
  - Static-site + optional-backend (graceful degradation) architecture.
- **Depth bar:** each decision is understandable and reversible by a future
  maintainer who wasn't there.
- **Sources:** requirements.md §5 + the two intent decisions from this session +
  git history + the old SDD's open questions.

- **Depth bar:** followable end-to-end to a running clone.
- **Sources:** README "Backend Setup", `supabase-setup.md`.

## Known issues surfaced during drafting

Documenting the system this exhaustively flushed out real bugs and inconsistencies
in the code (not in the spec). These are flagged 🐞 in their sections; collected
here so they aren't lost. Each needs a product decision: fix, or document-as-is.

1. **🐞 Dead auth-chip wiring on the home header** (§6a) — `site.js` references
   `#authButton`/`#userChip`/`#userEmail`/`#logoutButton`/`#adminLink`, none of
   which exist in `index.html`. The sign-in button, signed-in chip, logout and admin
   link never render; the auth dialog is reachable only via the `#login`/`#auth`
   hash. "Mi cuenta" is the only account entry point.
2. **🐞 `is_owner` is never actually set** (§8) — `admin-property.js` assigns owner by
   doing a client-side `UPDATE profiles SET is_owner=true`, but `profiles` has **no
   UPDATE RLS policy**, so the write is silently denied. Owners get `owner_id` on the
   property but never the `is_owner` flag → the owner portal stays locked. Correctness
   bug (fails closed, not an exposure).
3. **🐞 Server overlap check ignores expired holds** (§5) — if the 🔜 `request-booking`
   path is ever wired, its availability query doesn't filter expired holds, while the
   client does → a range the client offers could be rejected 409.
4. **🐞 Inconsistent overlap inclusivity in the client** (§5) — the home grid uses
   exclusive `<` (`rangesOverlap`) while the property page uses inclusive `<=`
   (`rangeHasConflict`) — a range that merely *touches* a block passes the grid but
   shows "conflict" on the detail page.
5. **🐞 Privacy page EN block unreachable** (§6a) — `privacy.html` ships hardcoded ES
   and EN blocks but never swaps them on the language toggle.
6. **🐞 World-readable `availability_blocks`** (§8) — the `using (true)` read policy
   exposes `user_id` *and* the admin `note` field to anonymous clients.
7. **⚠️ Two deploy workflows coexist** (§3) — `pages.yml` (GitHub Pages, canonical per
   README) and an Azure SWA workflow; only the Azure one runs `inject-config.js`.
   Confirm which is intended.
8. **⚠️ Nominatim usage-policy gaps** (§7) — client geocoding sends no descriptive
   User-Agent and no throttling, against OSM policy.
9. **⚠️ Open spam vectors** (§8) — `inquiries` and `owner_leads` allow anon insert
   (`with check (true)`) with no CAPTCHA/rate-limit.
10. **⚠️ Naming/scope inconsistencies** (§1/§2) — "1–12 months" (marketing) vs the
    enforced max of **11**; property type "house" vs the code's `home`.
